۱. معرفی پروژه

این برنامه وب‌سایت عمومی و پنل حساب کاربری یک پلتفرم فروش دوره‌های آنلاین کاراته است. امکانات اصلی آن عبارت‌اند از:

  • صفحه فرود فارسی/انگلیسی.
  • فهرست دوره‌ها و صفحه جزئیات هر دوره.
  • سبد خرید پایدار در مرورگر.
  • ثبت‌نام، ورود، خروج، بازیابی رمز با OTP و تغییر رمز.
  • پروفایل محافظت‌شده و قابل ویرایش.
  • مشاهده لینک دوره‌های خریداری‌شده.
  • تسویه‌حساب و هدایت/تأیید پرداخت به سبک درگاه زرین‌پال.
  • پنل مدیر برای ایجاد و حذف دوره و مشاهده دوره‌های یک کاربر.
  • عضویت در خبرنامه.

این مخزن شامل سرور بک‌اند، طرح پایگاه داده، سرویس ارسال ایمیل یا پیاده‌سازی درگاه پرداخت نیست. آدرس بک‌اند از NEXT_PUBLIC_BACKEND_URL خوانده می‌شود.

۲. راه‌اندازی سریع

پیش‌نیازها

  • Node.js نسخه ۲۰ LTS پیشنهاد می‌شود. Next.js 14 حداقل به Node.js 18.17 نیاز دارد.
  • npm؛ فایل package-lock.json در مخزن وجود دارد و مدیر بسته مرجع پروژه npm است.
  • Git.
  • بک‌اند جداگانه و قابل دسترس؛ قرارداد فعلی توسعه محلی http://localhost:3001 است.

نصب و اجرای محلی

git clone <repository-url>
cd karate-courses
npm ci

فایل .env.local را ایجاد کنید:

NEXT_PUBLIC_BACKEND_URL=http://localhost:3001

ابتدا بک‌اند و سپس فرانت‌اند را اجرا کنید:

npm run dev

مرورگر را در http://localhost:3000 باز کنید.

برای نصب تمیز و قابل تکرار، به‌ویژه در CI، از npm ci استفاده کنید. npm install را زمانی اجرا کنید که قصد تغییر وابستگی‌ها و lockfile را دارید.

بررسی سریع بعد از اجرا

۱. صفحه اصلی را باز کنید.
۲. به /courses بروید و نمایش دوره‌های بک‌اند را بررسی کنید.
۳. یک کاربر عادی ثبت‌نام کرده و وارد شوید.
۴. دوره‌ای را به سبد اضافه کرده و تسویه‌حساب را باز کنید.
۵. در /profile اطلاعات و دوره‌های خریداری‌شده را ببینید.
۶. با حساب مدیر وارد شده و /profile/admin را بررسی کنید.

کاربران عادی

کاربر عادی باید در /auth/register حساب خودش را بسازد و حساب مشترک از پیش آماده‌ای برای او وجود ندارد. فرم ثبت‌نام شامل نام، نام خانوادگی، ایمیل و رمز عبور است و پس از موفقیت به /auth/login هدایت می‌شود.

رفتار نقش‌ها

  • مهمان صفحه اصلی و دوره‌ها را می‌بیند و می‌تواند یک سبد محلی داشته باشد.
  • کاربر واردشده به پروفایل، تسویه‌حساب، تأیید پرداخت و دوره‌های خریداری‌شده دسترسی دارد.
  • مدیر در JWT مقدار isAdmin: true دارد و می‌تواند /profile/admin را باز کند.
  • صفحه جزئیات دوره برای مدیر عملیات حذف/مدیریت و برای کاربر عادی افزودن به سبد را نمایش می‌دهد.

۷. مسیرها و قوانین دسترسی

مسیر روش رندر دسترسی مسئولیت
/ Static عمومی صفحه فرود
/courses Force Static عمومی دریافت و نمایش دوره‌ها و خبرنامه
/courses/[courseId] Force Dynamic عمومی جزئیات و عملیات وابسته به نقش
/auth/login Static فقط مهمان ورود و دریافت کوکی از بک‌اند
/auth/register Static فقط مهمان ثبت‌نام کاربر عادی

/profile Dynamic/No-store واردشده ویرایش حساب، رمز و دوره‌های خریداری‌شده
/profile/admin Dynamic/No-store فقط مدیر ایجاد دوره و جستجوی دوره‌های کاربر
/payment/checkout Dynamic/No-store واردشده خلاصه سبد و آغاز پرداخت

src/middleware.ts برای /profile/:path*، /auth/:path* و /payment/:path* اجرا می‌شود:

  • نبود کوکی auth-token در پروفایل/پرداخت باعث هدایت به /auth/login است.
  • کاربر واردشده از /auth/* به /profile هدایت می‌شود.
  • JWT غیرمدیر از /profile/admin به /profile هدایت می‌شود.

۸. جریان‌های اصلی کاربر

ثبت‌نام و ورود

۱. کاربر عادی اطلاعات را با POST /register ارسال می‌کند.
۲. پس از موفقیت به /auth/login می‌رود.
۳. فرم ورود POST /login را با credentials فعال ارسال می‌کند.
۴. بک‌اند باید کوکی auth-token را تنظیم و شیء عمومی user را برگرداند.
۵. auth store نسخه‌ای از کاربر را در localStorage ذخیره می‌کند.
۶. Middleware کوکی را می‌خواند و رابط تعاملی از نسخه محلی کاربر استفاده می‌کند.

بازیابی رمز

۱. PUT /forget-password ایمیل را ارسال می‌کند.
۲. فرانت‌اند فعلی انتظار دارد پاسخ دارای OTP باشد و آن را در toast نشان می‌دهد.
۳. POST /validate-otp بدنه { OTP, email } را می‌فرستد.
۴. PUT /reset-password رمز جدید و تکرار آن را می‌فرستد.
۵. برنامه خروج را انجام داده و به ورود هدایت می‌کند.

در تولید، OTP باید از کانال جداگانه مانند ایمیل/SMS برسد و در پاسخ یا رابط نمایش داده نشود.

سبد و پرداخت

۱. جزئیات دوره با GET /user/fetch-course مالکیت قبلی را بررسی می‌کند.
۲. دوره خریداری‌نشده به سبد Zustand افزوده می‌شود.
۳. سبد با کلید cart در localStorage ماندگار است.
۴. تسویه POST /payment/checkout را با شناسه‌های رشته‌ای دوره می‌فرستد.
۵. بک‌اند paymentUrl می‌دهد و مرورگر کامل به آن هدایت می‌شود.
۶. درگاه به /payment/verify?Authority=...&Status=... بازمی‌گردد.
۷. در حالت Status=OK، فرانت POST /payment/verify را با authority می‌فرستد.
۸. سفارش تأییدشده و لینک خارجی دوره‌ها نمایش داده می‌شوند.

بک‌اند باید قیمت، مالکیت و امکان خرید را دوباره محاسبه کند؛ قیمت و شناسه ارسالی کلاینت قابل اعتماد نیستند.

مدیریت دوره

۱. با حساب مدیر وارد شوید.
۲. از پروفایل /profile/admin را باز کنید.
۳. عنوان، توضیح، قیمت عددی، تصویر base64 و لینک را با POST /create-course ارسال کنید.
۴. در صفحه جزئیات حذف را بزنید؛ DELETE /delete-course/:courseId اجرا می‌شود.
۵. جستجوی ایمیل، GET /admin/fetch-course/:email را برای مشاهده خریدهای کاربر فراخوانی می‌کند.

۹. مدیریت و ماندگاری وضعیت

Auth store

  • کلید localStorage: user.
  • وضعیت: { email, firstName, lastName } | null.
  • login شیء عمومی کاربر را ذخیره می‌کند.
  • logout ذخیره محلی را پاک، POST /logout را فراخوانی و صفحه اصلی را reload می‌کند.
  • کوکی را store تنظیم نمی‌کند؛ این کار بر عهده بک‌اند است.

Cart store

  • کلید localStorage: cart.
  • برای نمایش آسان، کل شیء دوره ذخیره می‌شود.
  • شناسه تکراری را نمی‌پذیرد و remove، clear و وضعیت hydration دارد.
  • سبد مخصوص همان مرورگر است و با بک‌اند همگام نمی‌شود.

Language store

  • کلید localStorage: language.
  • مقدار fa یا en.
  • فرهنگ فعال را با t در اختیار کامپوننت می‌گذارد.

الگوی hydration

localStorage هنگام رندر سرور وجود ندارد. Navbar، Cart، فرم‌های پروفایل، checkout و UserCourses تا پس از mount خروجی وابسته به مرورگر را به تأخیر می‌اندازند و placeholder/spinner نشان می‌دهند. برای وضعیت‌های مرورگری جدید همین الگو را رعایت کنید تا hydration mismatch ایجاد نشود.

۱۰. ارتباط با بک‌اند و قرارداد API

تمام درخواست‌ها از Axios موجود در src/services/httpService.ts استفاده می‌کنند:

  • base URL برابر NEXT_PUBLIC_BACKEND_URL.
  • قالب JSON.
  • withCredentials: true برای ارسال کوکی.
  • بدون timeout یا interceptor.

ApiClient<T> متدهای عمومی get، post، put و delete را برای یک endpoint ارائه می‌کند. Generic فعلی همه envelopeهای پاسخ را به‌طور کامل type-safe نمی‌کند.

endpointهای استنباط‌شده از فرانت‌اند

قرارداد زیر از نحوه مصرف فرانت استنباط شده و مرجع نهایی باید بک‌اند باشد.

متد endpoint درخواست پاسخ مورد انتظار فرانت
POST /register { firstName, lastName, email, password } { message }
POST /login { email, password } { message, user } و کوکی auth-token
POST /logout — { message } و حذف کوکی
PUT /forget-password { email } { OTP, message? }
POST /validate-otp { OTP: number, email } { message }
PUT /reset-password { newPassword, repeatPassword } { message }
PUT /profile { firstName, lastName, email } { message }؛ ممکن است توکن جدید لازم باشد
GET /fetch-course — Course[]
GET /fetch-course/:id — Course
POST /create-course { title, description, price, img, link } { message }
DELETE /delete-course/:id — { message }
GET /user/fetch-course کوکی Course[] خریداری‌شده
GET /admin/fetch-course/:email کوکی مدیر دوره‌های خریداری‌شده کاربر
POST /payment/checkout { courseIds: string[] } { message, paymentUrl }
POST /payment/verify { authority } { courses, transaction, message? }
POST /register-newsletter { email } { message }

الزامات بک‌اند

  • دریافت JSON.
  • CORS همراه credentials برای origin دقیق فرانت؛ در این حالت origin نمی‌تواند * باشد.
  • تنظیم کوکی دقیقاً با نام auth-token و Domain/Path/SameSite/Secure مناسب.
  • JWT شامل email، firstName، lastName و isAdmin بولی.
  • اعمال مستقل احراز هویت و مجوز مدیر در تمام APIهای محافظت‌شده.
  • اعتبارسنجی کامل ورودی و ترجیحاً خطا با ساختار { message }.
  • محاسبه مجدد مبلغ و تأیید authority در سرور.

هدرهای Access-Control-Allow-* که اکنون به درخواست Axios افزوده شده‌اند، CORS مرورگر را تنظیم نمی‌کنند؛ CORS باید در پاسخ بک‌اند پیکربندی شود.

۱۱. مدل داده و اعتبارسنجی

دوره عمومی

{

  title: string;       // بین ۵ تا ۵۰ کاراکتر
  description: string; // بین ۲۰ تا ۱۵۰ کاراکتر در Course

مدل ایجاد دوره link را اضافه می‌کند و توضیح تا ۱۰۰۰ کاراکتر می‌پذیرد. تصویر می‌تواند data URI، آدرس HTTP(S)، مسیر /public یا base64 خام باشد. helper با پیشوند base64 نوع PNG، WebP یا GIF را تشخیص می‌دهد و در غیر این صورت JPEG فرض می‌کند. لینک بدون scheme با https:// کامل می‌شود.

claimهای کاربر/JWT

{

  firstName: string;
  lastName: string;

قانون رمز عبور

Regex مشترک ^(?=.*[a-z])(?=.*\d).{8,}$ است:

  • حداقل ۸ کاراکتر.
  • حداقل یک حرف کوچک انگلیسی.
  • حداقل یک رقم.

با وجود متن «یک حرف»، رمز فقط با حرف بزرگ یا فقط حروف فارسی در پیاده‌سازی فعلی پذیرفته نمی‌شود.

۱۲. احراز هویت، مجوز و امنیت

مدل فعلی

  • وجود auth-token نشانه دسترسی به مسیر است.
  • Middleware توکن را برای خواندن isAdmin decode می‌کند، ولی امضای آن را بررسی نمی‌کند.
  • user در localStorage فقط برای نمایش تعاملی است.
  • تمام درخواست‌های Axios credentials دارند.

قواعد ضروری تولید

  • بک‌اند در هر API محافظت‌شده امضا، انقضا، issuer/audience لازم و نقش JWT را بررسی کند.
  • کوکی HttpOnly، در HTTPS دارای Secure و با سیاست آگاهانه SameSite باشد.
  • در صورت امکان درخواست cross-site دارای کوکی، حفاظت CSRF اضافه شود.
  • هرگز isAdmin رابط را مبنای مجوز قرار ندهید.
  • URL و اندازه/نوع تصویر base64 در سرور بررسی شود.
  • OTP در تولید به کلاینت برگردانده نشود.
  • برای ورود و OTP rate limit و پیام‌های ضد account enumeration در نظر گرفته شود.
  • رمز حساب آزمایشی پیش از انتشار عمومی عوض شود.
  • شناسه/راز پرداخت فقط در بک‌اند یا secret store باشد. هر NEXT_PUBLIC_* در bundle مرورگر قرار می‌گیرد.

بهداشت secret در مخزن

در حال حاضر .env ریشه توسط Git دنبال می‌شود و .gitignore فقط .env*.local را نادیده می‌گیرد. هر مقدار واقعی که commit شده را افشاشده در نظر بگیرید: secretها را rotate کنید، در صورت نیاز از تاریخچه حذف کنید، در توسعه از .env.local و در تولید از secretهای میزبان استفاده کنید. فقط یک .env.example بی‌خطر باید به‌عنوان الگو commit شود.

۱۳. چندزبانه‌سازی

فرهنگ‌ها در src/lang/fa.ts و src/lang/en.ts هستند و useLanguageStore() مقدار t و زبان ماندگار را ارائه می‌دهد.

پوشش فعلی کامل نیست:

  • صفحه فرود، navbar، تعدادی از فیلدهای پروفایل و عملیات عمومی از فرهنگ استفاده می‌کنند.
  • رشته‌های زیادی در دوره، checkout، admin، بازیابی رمز، footer، toast و validation هنوز فارسی hard-code هستند.
  • metadata، مقدار lang سند و RTL بودن Toastify با انتخاب انگلیسی همچنان فارسی می‌ماند.

برای متن جدید، کلید یکسان را به هر دو فرهنگ اضافه و از t استفاده کنید. پیاده‌سازی کامل‌تر باید lang، dir، metadata و جهت toast را نیز بر اساس locale تغییر دهد؛ مسیرهای localeدار یا state قابل خواندن در سرور گزینه مناسبی هستند.

۱۴. رابط کاربری، استایل و فایل‌های ثابت

  • پروفایل، مدیر و checkout به‌صورت dynamic/no-store اجرا می‌شوند.
  • تأیید پرداخت query string را در کلاینت و داخل Suspense می‌خواند.
  • داده مخصوص مرورگر بعد از mount hydrate می‌شود.

اثر static بودن فهرست دوره

اگر بک‌اند در زمان build خاموش باشد، catch صفحه فهرست خطا را می‌گیرد و آرایه خالی رندر می‌کند؛ بنابراین:

  • CI ممکن است ECONNREFUSED نشان دهد ولی موفق تمام شود.
  • deploy ساخته‌شده بدون دسترسی به بک‌اند می‌تواند فهرست دوره خالی داشته باشد.
  • تغییر دوره تا build/revalidation بعدی دیده نمی‌شود.

اگر تازگی داده ضروری است، dynamic rendering، revalidation زمان‌دار یا on-demand revalidation پس از عملیات مدیر را بررسی کنید.

۱۹. تست و بررسی

در حال حاضر تست خودکار unit، integration یا end-to-end در مخزن وجود ندارد. حداقل این موارد دستی بررسی شوند:

  • رندر عمومی در موبایل و دسکتاپ.
  • تعویض فارسی/انگلیسی و ماندگاری بعد از reload.
  • validation ثبت‌نام و redirect موفق.
  • کوکی ورود، خروج و redirect مسیرهای auth.
  • هدایت مهمان از profile/payment و غیرمدیر از admin.
  • ویرایش پروفایل و تغییر رمز.
  • فهرست/جزئیات، انواع تصویر، جلوگیری از سبد تکراری و remove/clear.
  • checkout خالی و پر.
  • لغو پرداخت، query ناقص، خطای verify و پرداخت موفق.
  • ایجاد/حذف مدیر و جستجو با ایمیل.
  • رفتار هنگام خاموش بودن بک‌اند.

ابزار پیشنهادی آینده: Vitest و React Testing Library برای unit/component، MSW برای mock API و Playwright برای جریان نقش‌ها و پرداخت.

۲۰. رفع اشکال

دوره‌ها خالی هستند یا build خطای ECONNREFUSED localhost:3001 دارد

  • بک‌اند را روی 3001 اجرا یا NEXT_PUBLIC_BACKEND_URL را اصلاح کنید.
  • پس از تغییر environment فرانت را restart کنید.
  • در تولید، بک‌اند را هنگام prerender قابل دسترس کنید یا روش رندر catalog را عوض کنید.

ورود موفق است ولی مسیر محافظت‌شده به login برمی‌گردد

  • در پاسخ بررسی کنید کوکی دقیقاً با نام auth-token تنظیم شده باشد.
  • Domain، Path، Expiry، SameSite و Secure را بررسی کنید.
  • credentials درخواست و CORS origin دقیق بک‌اند را بررسی کنید.

خطای CORS

  • CORS را در بک‌اند اصلاح کنید؛ هدر درخواست کلاینت مجوز ایجاد نمی‌کند.
  • برای credentials از Access-Control-Allow-Origin: * استفاده نکنید.

مسیر مدیر به پروفایل برمی‌گردد

  • JWT صادرشده باید isAdmin: true بولی داشته باشد.
  • مطمئن شوید کوکی دارای توکن فعلی و برای middleware قابل دیدن است.
  • بدون توجه به middleware، مجوز API مدیر را در بک‌اند اعمال کنید.

سبد/کاربر/زبان با تأخیر دیده می‌شود

این تأخیر hydration localStorage عادی است. اگر هرگز نمایش داده نشد، JSON خراب localStorage و console مرورگر را بررسی کنید.

خطای permission در .next/cache/eslint

  • مطمئن شوید build یا lint دیگری از .next استفاده نمی‌کند.
  • process نگهدارنده فایل را ببندید و دوباره lint بگیرید.
  • build و lint را هم‌زمان اجرا نکنید؛ هر دو در .next می‌نویسند.

خطای پرداخت

  • callback باید هر دو Authority و Status را داشته باشد.
  • فقط Status=OK باعث verify بک‌اند می‌شود.
  • /payment/verify باید { authority } را بپذیرد، با درگاه بررسی کند و شکل transaction/course مورد انتظار را برگرداند.

۲۱. محدودیت‌ها و نکات نگهداری

  • فرانت و بک‌اند بدون قرارداد API تولیدشده/مشترک به response shapeها وابسته‌اند.
  • Middleware فقط JWT را decode می‌کند؛ مرز امنیتی نیست.
  • نسخه user در localStorage ممکن است با کوکی همگام نباشد.
  • پس از ویرایش پروفایل، بک‌اند ممکن است نیاز داشته باشد claimهای JWT را تازه کند؛ در غیر این صورت رندر سرور اطلاعات قدیمی می‌بیند.
  • بیشینه توضیح در schema ایجاد دوره ۱۰۰۰ و در schema عمومی دوره ۱۵۰ است.
  • catalog ثابت ممکن است قدیمی یا بدون خطای نهایی خالی build شود.
  • بعضی toastهای cart مستقیماً فرهنگ فارسی را استفاده می‌کنند.
  • پوشش ترجمه و تغییر جهت سند کامل نیست.
  • OTP فعلاً در پاسخ قابل مشاهده فرانت انتظار می‌رود.
  • timeout، retry، نرمال‌سازی متمرکز خطا، error boundary و تست خودکار وجود ندارد.
  • ZARINPAL_MERCHANT_ID توسط فرانت خوانده نمی‌شود و باید تنظیم بک‌اند باشد.
  • سیاست .env ردیابی‌شده نیازمند امن‌سازی است.

این موارد توصیف وضعیت فعلی‌اند و الزام به حفظ رفتار نامناسب نیستند.

۲۲. چک‌لیست مشارکت

  • متن را هم‌زمان به فارسی و انگلیسی اضافه کنید.

  • حتی با check رابط و middleware، مجوز را در بک‌اند اعمال کنید.
  • موبایل و دسکتاپ را بررسی کنید.
  • رفتار مهمان، کاربر و مدیر را بررسی کنید.
